\section{Постановка задачи}

Алгоритмы достижимости с ограничениями в виде контекстно-свободных языков, основанные на операциях линейной алгебры и использующиеся для статического анализа кода, могут быть основаны на произведении Кронекера~\cite{orachev2020}. При вычислении произведения Кронекера для разреженных матриц возникает проблема избыточного потребления памяти: результирующая матрица содержит большое число нулевых элементов, которые требуют выделения большого объема памяти. В актуальной версии библиотеки SuiteSparse:GraphBLAS~\cite{graphblas} фильтрация ненулевых элементов выполняется после построения результата, что приводит к высокому пиковому потреблению памяти и ограничивает применимость алгоритмов на больших графах.

\textbf{Целью работы является} добавление фильтрации нулевых элементов до этапа выделения памяти для результирующей матрицы в алгоритм произведения Кронекера с целью снижения пикового потребления памяти.

Результаты работы необходимы для повышения практической применимости алгоритмов, основанных на произведении Кронекера, для анализа больших графов.

\section{Описание предлагаемого решения}

Основой для предлагаемого решения послужил алгоритм вычисления произведения Кронекера с ранней фильтрацией элементов, разработанный В.~А.~Кутуевым~\cite{kutuev2023}. Суть данного подхода заключается в предварительной оценке количества ненулевых значений результирующей матрицы и выделении памяти только под них. Таким образом, выделение памяти выполняется только под значащие элементы результата, что позволяет избежать избыточного потребления памяти.

В рамках данной работы этот алгоритм был адаптирован и перенесен на актуальную версию библиотеки SuiteSparse:GraphBLAS (в исходной работе использовалась версия 7.4.3). В частности, была добавлена поддержка JIT-компиляции, которая отсутствовала в предыдущих релизах библиотеки. Это позволило сократить время вычисления произведения по сравнению со стандартной реализацией.

\subsection{Реализация алгоритма}

Для реализации ранней фильтрации используется двухпроходный алгоритм.
\begin{enumerate}
    \item \textbf{Первый проход:} подсчёт количества ненулевых элементов для каждого выходного вектора с использованием временных массивов.
    \item \textbf{Выделение памяти:} создание выходной матрицы $C$ с точным размером \texttt{cnz} вместо верхней границы \texttt{cnzmax}.
    \item \textbf{Второй проход:} запись результата в выходную матрицу $C$ с учётом пропуска нулевых элементов.
\end{enumerate}

\subsection{Изменение логики распараллеливания}

В процессе реализации появилась проблема в логике распараллеливания для случая использования селектора (механизма фильтрации элементов, определяющего, какие значения должны быть записаны в результат, посредством побайтовой проверки на равенство нулю) совместно с гиперразреженными матрицами (матрицами с преобладающим количеством нулевых элементов). 

Исходная реализация использовала поэлементное распараллеливание, при котором работа распределялась между потоками по отдельным ненулевым элементам результирующей матрицы $C$. Данный подход опирался на предположение, что каждый ненулевой элемент матрицы $A$ генерирует ровно \texttt{bknz} записей в матрице $C$, где \texttt{bknz}~--- количество ненулевых элементов в соответствующем столбце матрицы $B$. Для алгоритма с фильтрацией это предположение нарушается: селектор отбрасывает часть вычисленных значений, и реальное количество записей становится непредсказуемым. 

Хотя данная проблема в принципе возникает для любых разреженных матриц при использовании селектора, наиболее критично она проявляется именно для гиперразреженных матриц. Это обусловлено тем, что в гиперразреженном формате хранятся только непустые векторы, и ошибка в расчёте смещений приводит к обращению к совершенно неверным векторам.

Для решения этой проблемы в оптимизированной версии осуществлён переход к векторному распараллеливанию: работа распределяется между потоками не по отдельным элементам, а по целым столбцам (векторам) результирующей матрицы $C$. Данный подход полностью устранил необходимость в сложных вычислениях смещений. Стоит отметить, что поэлементное распараллеливание позволяет более равномерно распределить нагрузку между потоками, вследствие чего может уменьшаться время вычисления произведения Кронекера.

\subsection{Особенности работы с JIT-компиляцией}

Библиотека SuiteSparse:GraphBLAS поддерживает два основных пути исполнения операций, выбор которых зависит от типа используемого оператора.

При использовании стандартных встроенных в библиотеку операций активируется механизм JIT-компиляции. В этом случае формируется JIT-ядро~--- специализированный участок кода, оптимизированный исключительно под конкретную операцию и типы данных. Это позволяет исключить накладные расходы на проверку типов и косвенные вызовы, обеспечивая максимальную производительность.

При использовании пользовательских операций JIT-компилятор не может сгенерировать полностью специализированное ядро. В этом случае система выполняет возврат к использованию generic-ядра, которое представляет собой единую реализацию алгоритма, способную обрабатывать любые типы данных и пользовательские операторы.

В одной из версий реализации алгоритма этап подсчета ненулевых элементов выполнялся исключительно как inline-код, что исключало возможность применения механизмов JIT-компиляции для данной части вычислений. Однако логика была модифицирована таким образом, чтобы обеспечить корректный вызов JIT-компилятора и на этапе подсчета.

В результате сравнительного анализа было установлено, что использование JIT-компиляции для подсчета ненулевых элементов приводит к незначительному увеличению затрат памяти по сравнению с чисто inline-реализацией. Однако этот расход памяти компенсируется существенным выигрышем во времени выполнения.

Реализация решения осуществлялась с использованием языка программирования C, библиотеки SuiteSparse:GraphBLAS.

\section{Эксперименты}

Проведённое экспериментальное исследование алгоритма показало, что на данных, характерных для различных задач статического анализа кода (например, alias analysis и points-to анализ), удаётся снизить пиковое потребление памяти, но это происходит за счёт увеличения времени выполнения операции по сравнению со стандартной реализацией, при сохранении корректности результатов и стабильности работы на различных наборах входных данных (графы программ и рекурсивные автоматы~--- способ описания контекстно-свободного языка). Экспериментальное исследование заключалось в проведении 40 запусков алгоритма на входном наборе матриц, полученных в результате работы алгоритма статического анализа на реальных программах. Результаты замеров сгруппированы по названию анализируемого проекта (ось X графиков). На графиках время неоптимизированной версии считается как сумма времени на вычисление Кронекера и время на фильтрацию ненулевых элементов. 

Рассматривались две конфигурации оптимизированного алгоритма:
\begin{enumerate}
    \item версия без JIT-компиляции (generic kernels);
    \item версия с JIT-компиляцией.
\end{enumerate}

\paragraph{Среда проведения экспериментов.}

12th Gen Intel(R) Core(TM) i3-1215U.

\subsection{Анализ результатов}

Эксперименты показали значительное снижение пикового потребления памяти для обеих версий оптимизированного алгоритма.
\begin{itemize}
    \item Версия без JIT: сокращение потребления памяти в 2.11--2.38 раза.
    \item Версия с JIT: сокращение потребления памяти в 2.09--2.35 раза.
\end{itemize}

Наибольшее снижение памяти наблюдается на графе sunflow.

При этом оптимизация приводит к увеличению времени выполнения, что является ожидаемым компромиссом.
\begin{itemize}
    \item Версия без JIT: ускорение 0.70--0.74x (замедление примерно на 36--42\%).
    \item Версия с JIT: ускорение 0.96--0.99x (замедление примерно на 1--4\%).
\end{itemize}

Применение JIT-компиляции позволило существенно сократить разрыв со стандартной реализацией, в абсолютных значениях времени версия с JIT работает значительно быстрее версии без JIT (рис.~\ref{fig:v3}).

Увеличение времени относительно стандартной реализации обусловлено необходимостью двух проходов по данным вместо одного.

\subsection{Выводы из экспериментов} 
Предложенная оптимизация успешно решает поставленную задачу снижения пикового потребления памяти, что особенно важно для анализа больших графов программ. Увеличение времени выполнения является приемлемым компромиссом, так как во многих сценариях статического анализа ограничение по памяти является более критичным фактором, чем время выполнения.

\section{Заключение}

В ходе выполнения работы получены следующие результаты.

\begin{itemize}
    \item Добавлена фильтрация нулевых элементов на этапе вычисления в алгоритм произведения Кронекера для разреженных матриц в SuiteSparse:GraphBLAS.
    \item Реализован двухпроходный алгоритм с предварительным подсчётом ненулевых элементов и выделением памяти только под значащие элементы результата.
    \item Достигнуто снижение пикового потребления памяти произведением Кронекера в 2.1--2.4 раза на различных наборах данных из области статического анализа кода.
    \item Алгоритм адаптирован к актуальной версии библиотеки SuiteSparse:GraphBLAS.
    \item Проведённое экспериментальное исследование подтвердило эффективность оптимизации по памяти при приемлемом увеличении времени выполнения.
\end{itemize}

Решение не является финальным и открывает направления для дальнейшей работы:
\begin{itemize}
    \item дальнейшая оптимизация производительности с целью сведения временных затрат к уровню стандартной реализации библиотеки;
    \item добавление возможности отбирать ненулевые элементы по пользовательскому критерию вместо использующейся сейчас побайтовой проверки на 0.
\end{itemize}
